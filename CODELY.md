## Codely Added Memories
- Regras de operação obrigatórias para o projeto DisciplinaApp: 1) Escopo Controlado — executar apenas o solicitado, sem funcionalidades extras, sem antecipar fases futuras. 2) Arquivos — nunca criar além dos autorizados, nunca modificar fora da lista aprovada, solicitar aprovação para novos arquivos. 3) Arquitetura — DisciplinaApp = Produto Principal, Kairos = IA Central. Camadas futuras (Coach/Business/Ops) não implementar sem aprovação. 4) Implementação Incremental — pequenos lotes, concluir fase antes de iniciar próxima, validar cada etapa. 5) Evidências Obrigatórias — ao finalizar apresentar: arquivos modificados, criados, erros corrigidos, builds executados, resultados dos testes, pendências. 6) Prevenção de Loops — não gerar código em massa, não criar estruturas futuras "para já deixar pronto", não criar placeholders desnecessários. 7) Dúvidas — parar, explicar, solicitar aprovação. Nunca assumir decisões arquiteturais sem aprovação explícita.
- REGRA OPERACIONAL OBRIGATÓRIA — VALIDAÇÃO PRÉ-IMPLEMENTAÇÃO: Para todo lote futuro: 1) Ler arquivos reais envolvidos. 2) Confirmar contratos reais da API e tipos. 3) Listar todos os arquivos que serão criados, modificados ou removidos. 4) Identificar riscos e dependências. 5) Definir critérios de conclusão verificáveis. 6) Definir evidências obrigatórias (build, lint, typecheck, testes). 7) Somente após auditoria prévia iniciar implementação. PROIBIÇÕES: Não assumir contratos. Não inferir payloads. Não criar código baseado em estimativas. Não expandir escopo sem aprovação explícita.
- REGRA PERMANENTE DO PROJETO — CONTRATO + EVIDÊNCIA OBRIGATÓRIOS: Nenhuma integração será considerada concluída apenas por compilar. Todo lote que integrar backend, auth, IA ou serviços externos deverá apresentar obrigatoriamente: 1) Contrato da integração, 2) Evidência de execução real, 3) Build, 4) Lint, 5) Typecheck. Sem esses cinco itens, o lote permanece pendente.
- DÍVIDA TÉCNICA REGISTRADA: api.ts depende do rewrite do Next.js e utiliza URLs relativas (/api/*). Estratégia aceita para o MVP atual client-side. Revisar apenas se houver SSR/Server Components autenticados no futuro.
